# Messaging Skill Specification

## Purpose

Define a file-based, inbox-driven protocol for agent-to-agent communication within a shared
workspace. Four tools implement the protocol:

- **`init`** — an agent invokes this once on startup to register its identity in the shared
  inbox space. Creates the inbox directory, archive subdirectory, and signal file. Fails
  if the name is already taken, providing an atomic name-claim primitive.

- **`post`** — an agent invokes this to post a message to another agent's inbox. The tool
  owns all mechanics: timestamp generation, nonce, atomic write. The agent supplies only
  intent: recipient, subject, and body.

- **`drain`** — an agent invokes this to collect and consume messages from its own inbox.
  The tool claims, reads, and archives each message, returning the message content to the
  caller. The agent processes the returned content.

- **`status`** — a lightweight, read-only probe. An agent wires this directly to its
  Monitor as the signal-change callback. `status` counts unclaimed messages in the inbox
  and outputs the pending count. Does not claim, read, or modify any files.

Agents are not expected to generate timestamps, produce nonces, perform atomic renames, or
manage file state directly. Those responsibilities belong to the tools.

This is the root specification for a suite of messaging sub-skills. v1 establishes the
envelope format, naming rules, ordering contract, contention model, and the `init`, `post`,
`drain`, and `status` tool interfaces.

## Scope

Applies to any pair of agents sharing a common workspace with read/write filesystem access.

In scope:

- Inbox folder structure and location
- Message filename format
- Message file envelope (frontmatter + body)
- `init` tool interface and behavior, including name-claim atomicity
- `post` tool interface and behavior, including atomicity
- `drain` tool interface and behavior, including claim and archive
- `status` tool interface and behavior
- Ordering guarantees
- Collision avoidance
- Contention avoidance
- Monitoring behavior

Out of scope:

- Transport other than a shared local filesystem
- Encryption or access control beyond filesystem permissions
- Message routing through intermediaries (broker, relay)
- Guaranteed delivery or retry protocol
- Message expiry, TTL, or inbox cleanup (deferred to a sub-skill)
- Agent discovery or name registry

## Definitions

- **Inbox**: the directory `.inbox/<agent-name>/` relative to the workspace root, owned by
  the named agent as its exclusive read target.
- **Archive**: the directory `.inbox/<agent-name>/archive/`, where drained messages are
  moved after being read. Archive may be purged freely; it has no protocol role.
- **Agent name**: the canonical, stable identifier for an agent. Lowercase, hyphen-separated
  (kebab-case). Examples: `curator`, `code-review-worker`, `overseer`.
- **Message file**: a single `.json` file placed in a recipient's inbox, representing exactly
  one message.
- **Poster**: the agent invoking the `post` tool to place a message in another agent's inbox.
- **Recipient**: the agent whose inbox receives the message.
- **Init tool**: the tool an agent invokes once on startup to register its identity.
  Creates the inbox directory, archive subdirectory, and signal file. Uses atomic
  directory creation to guarantee only one agent claims a given name. Supports `--force`
  for agent restart (reclaim without failing).
- **Post tool**: the tool that agents invoke to post a message. The tool generates the
  timestamp, nonce, envelope, and performs the atomic write.
- **Drain tool**: the tool that agents invoke to collect messages from their own inbox.
  The tool claims, reads, archives each message, and returns the message content to the
  caller.
- **Status tool**: the tool that an agent wires directly to its Monitor as the
  signal-change callback. The tool counts unclaimed message files in the inbox and outputs
  the pending count. Read-only; does not claim, read, or modify any files.
- **Envelope**: the YAML frontmatter block at the top of a message file carrying routing
  metadata.
- **Body**: the Markdown content below the frontmatter block, containing the message payload.
- **Nonce**: a short random string generated by the post tool and included in the message
  filename to ensure uniqueness.
- **Monitor**: a filesystem watch capability the recipient uses to observe its inbox for
  new files. The specific mechanism (filesystem watcher, polling, VS Code file watcher) is
  implementation-defined.
- **Signal file**: the file `.inbox/<agent-name>/.signal`, used exclusively as a wake
  trigger. The post tool writes this file after each successful message write. The
  recipient's Monitor watches this file for changes. Signal file content is not meaningful;
  any change to it means "at least one new message may be pending."
- **Claim**: an exclusive reservation on a message file, made by the drain tool before
  reading. A claimed file is not readable by a concurrent drain invocation. See
  `implementation.md` for the mechanism.
- **Unclaimed**: a message file with the `.md` extension that has not yet been claimed.
- **Archived**: a claimed message that has been moved to the archive folder after reading.
  Archiving is the terminal state for successfully drained messages.

## Content Modes

This skill operates in one mode: **runtime protocol**. Agents apply these rules directly
without dispatching a sub-agent. There is no dispatch variant in v1.

## Requirements

### Init Tool Interface

R0. An agent MUST invoke the init tool once on startup, before draining or entering
    the monitoring loop.

R0a. The init tool MUST accept the following inputs:

    - `--name` (required): the agent's canonical name to register
    - `--workspace` (optional): workspace root path; defaults to `$PWD`
    - `--force` (optional): reclaim an existing inbox without failing

R0b. The init tool MUST use atomic directory creation (fail-if-exists semantics) to
     create `.inbox/<name>/`. If the directory already exists and `--force` is not set,
     the tool MUST exit non-zero with an error naming the taken inbox.

R0c. On successful registration, the init tool MUST create the `archive/` subdirectory
     and write the signal file. Signal file write failure on init is fatal (unlike `post`
     where it is tolerated); the watcher cannot attach without a signal file.

R0d. With `--force`, the init tool MUST create the inbox if absent, skip creation if
     present, and always write the signal file. It MUST NOT modify existing messages.

R0e. The init tool MUST exit with a zero exit code on success and a non-zero exit code
     on failure. On failure, a human-readable error message MUST be written to stderr.

R0f. The init tool MUST provide a `--help` flag that documents all inputs and behavior.

### Inbox Structure

R1. An agent's inbox MUST be located at `.inbox/<agent-name>/` relative to the workspace
    root.

R2. The `<agent-name>` path segment MUST be lowercase and hyphen-separated (kebab-case).
    It MUST match the agent's canonical name exactly.

R3. The post tool MUST NOT post a message to the poster's own inbox.

R4. The post tool MUST create the recipient's inbox directory if it does not already exist,
    before writing the message file. This handles posting to an agent that has not yet
    called `init`; the recipient's init call will succeed if its inbox directory is absent,
    or use `--force` if it already exists.

### Message Filename Format

R5. The post tool MUST generate message filenames following this pattern:

    <utc-timestamp>-<nonce>.json

    where `<utc-timestamp>` is the post time in compact ISO 8601 UTC format
    `YYYYMMDDTHHmmssZ`, and `<nonce>` is 6 to 8 lowercase hex characters.

    Example: `20260508T143022Z-a3f91b.json`

R6. The post tool MUST generate the nonce from a cryptographically random source.
    Sequential counters, PIDs, or deterministic values MUST NOT be used as the nonce.

R7. If the intended filename already exists in the target inbox, the post tool MUST
    regenerate the nonce and retry until a unique filename is produced.

### Message File Format

R8. Every message file MUST be a valid JSON object with exactly these fields:

    - `from`: the posting agent's canonical name (string)
    - `to`: the receiving agent's canonical name (string)
    - `sent`: the UTC post timestamp in ISO 8601 format (string)
    - `subject`: a short, human-readable description of the message intent (string)
    - `body`: the message content (string)

R9. The JSON object MUST be compact (no pretty-printing required) and UTF-8 encoded.
    String values MUST be properly JSON-escaped.

R10. A message file MUST NOT be modified after it has been written to the inbox.

### Post Tool Interface

R11. The post tool MUST accept the following inputs:

    - `--from` (required): the posting agent's canonical name
    - `--to` (required): the recipient agent's canonical name
    - `--subject` (required): a short description of the message intent
    - `--body` (required): the message body text

R12. The post tool MUST write message files atomically. A partially written file MUST
     never be visible in the inbox. Agents MUST NOT write inbox files directly; they MUST
     use the post tool. See `implementation.md` for the recommended technique.

R13. After successfully writing the message file, the post tool MUST write the signal
     file (`.inbox/<agent-name>/.signal`). If the signal write fails, the post tool MUST
     still exit zero; the message is delivered and drain will collect it on the next
     signal or startup.

R14. The post tool MUST exit with a zero exit code on success and a non-zero exit code on
     failure. On failure, a human-readable error message MUST be written to stderr.

R15. The post tool MUST provide a `--help` flag that documents all inputs and behavior.

### Monitoring

R16. The recipient MUST monitor the signal file (`.inbox/<own-name>/.signal`) for changes.
     Monitoring MUST begin before the agent enters its idle state.

R17. The signal file is edge-triggered: it changes only when a new message is posted.
     The recipient MUST NOT assume the number of new messages from the number of signal
     events. One signal event may correspond to multiple messages; drain always sweeps all
     pending messages regardless.

R17a. On any change event on the signal file, the recipient SHOULD invoke the status
      tool to report the pending message count. The recipient MUST then invoke the drain
      tool and process all returned messages.

R17b. After processing completes, the recipient MUST invoke drain once more before
      resuming the watch. If that final drain returns any messages, the recipient MUST
      process them and drain again. The recipient MUST repeat until drain returns empty.
      This drain-to-quiescence loop ensures messages that arrived during processing are
      not stranded by a missed signal event.

R18. On startup, the recipient MUST invoke the drain tool before entering its normal
     workflow. Messages posted while the agent was offline MUST be processed as if they
     had just arrived.

### Drain Tool Interface

R19. The drain tool MUST accept the following inputs:

    - `--inbox` (required): the agent name whose inbox to drain

R20. The drain tool MUST enumerate all unclaimed message files (files matching `*.json`
     directly inside the inbox, excluding any files in subdirectories such as `archive/`).

R21. The drain tool MUST process unclaimed files in ascending filename order
     (lexicographic sort).

R22. For each unclaimed file, the drain tool MUST:

    a. Claim the file before reading it. If the claim fails (file gone or already claimed
       by a concurrent drain), skip the file silently.
    b. If the claim succeeds, read the file content.
    c. Move the file to `.inbox/<agent-name>/archive/<name>.json`.
    d. Collect the parsed JSON object into the result list.

    See `implementation.md` for the claim mechanism.

R23. The drain tool MUST NOT halt on a single unreadable or malformed file. It MUST
     continue to the next file and report the failure on stderr.

R24a. The drain tool MUST output a JSON array of message objects to stdout. The array
      MUST be in ascending timestamp order. An empty inbox MUST produce `[]`.

R24b. The drain tool MUST exit with a zero exit code when draining completes (even if some
      files were skipped). A non-zero exit code indicates a tool-level failure.

R25. The drain tool MUST provide a `--help` flag that documents all inputs and behavior.

### Status Tool Interface

R_N1. The status tool MUST accept the following inputs:

    - `--inbox` (required): the agent name whose inbox to check

R_N2. The status tool MUST count unclaimed message files — files matching `*.json` directly
      inside the inbox directory, excluding any files in subdirectories such as `archive/`.
      The status tool MUST NOT claim, read, move, or modify any file.

R_N3. The status tool MUST output exactly one line to stdout:

        [<utc-timestamp>]: <N> messages waiting

      where `<utc-timestamp>` is the current UTC time in ISO 8601 format and `<N>` is
      the count of unclaimed message files. Output MUST be written even when N is zero.

R_N4. The status tool MUST exit with a zero exit code on normal completion. A non-zero
      exit code indicates a tool-level failure (e.g., inbox directory inaccessible).

R_N5. The status tool MUST provide a `--help` flag that documents all inputs and behavior.

R26. Message order is significant within an inbox. The drain tool MUST process unclaimed
     messages in ascending filename order (lexicographic sort on the filename string).

R27. Because filenames begin with a UTC timestamp, ascending filename order is equivalent
     to ascending post-time order, provided poster clocks are synchronized to a common
     time source.

R28. Messages from different posters may interleave in the inbox based on their timestamps.
     Per-poster ordering is preserved only to the extent that a single poster generates
     monotonically increasing timestamps between posts.

## Constraints

C1. Atomicity guarantees for file writes depend on operating system primitives. Network-
    mounted filesystems (NFS, SMB without appropriate configuration) may not provide these
    guarantees. Deployment on such filesystems is not supported in v1. See
    `implementation.md` for platform-specific guidance.

C2. v1 does not define message expiry or TTL. The `archive/` folder may be purged freely
    by the workspace operator; it has no protocol role after messages are drained.

C3. An inbox is intended for a single drain agent. Multiple agents draining the same inbox
    is not a supported configuration in v1. The claim-by-rename mechanism reduces (but does
    not eliminate) the risk if violated.

C4. v1 provides no delivery guarantee. If a recipient is permanently offline or its inbox
    is inaccessible, messages accumulate without retry or notification.

C5. v1 defines no security model. Any agent or process with filesystem access can read,
    post to, or drain any inbox. Agents should be aware that inbox isolation is
    conventional, not enforced.

## Behavior

### Posting a Message

Agent flow:

1. Agent decides to post a message (has recipient, subject, and body ready).
2. Agent invokes: `post --from <self> --to <recipient> --subject <subject> --body <body>`
3. Post tool generates timestamp and nonce, assembles envelope, writes atomically.
4. Post tool exits zero on success; agent checks exit code and treats non-zero as failure.

Agent responsibility ends at step 2. The tool owns all file mechanics.

### Idle Monitoring Loop

When the recipient has no pending tasks, it enters a monitoring loop:

1. Watch `.inbox/<own-name>/.signal` for change events.
2. On any change event:
   a. Optionally invoke `status --inbox <own-name>` to report the pending count.
   b. Invoke `drain --inbox <own-name>`. Process all returned messages (validate
      frontmatter, process body; log and continue on failure).
   c. Invoke drain again. If it returns messages, process them and repeat.
   d. Continue until drain returns empty (drain-to-quiescence, R17b).
3. Return to watching.

### Startup Check

On startup, the agent MUST:

1. Invoke `init --name <own-name>` (or `init --name <own-name> --force` on restart).
2. Drain the inbox once to collect any messages posted while offline.
3. Enter the monitoring loop.

### Malformed Message

If a message file cannot be parsed (missing frontmatter, invalid YAML, empty body), the
drain tool reports the failure on stderr and continues. The file is moved to `archive/`
regardless. The agent logs the failure and continues; it MUST NOT halt.

## Defaults and Assumptions

- The workspace root is a shared directory agreed upon by all participating agents.
- All participating agents have read/write access to the `.inbox/` subtree.
- Agent names are stable and known to all posters. There is no dynamic name resolution.
- System clocks are synchronized well enough that timestamp ordering is meaningful at
  one-second granularity. Sub-second ordering between different machines is not guaranteed.
- The nonce provides sufficient uniqueness when two posters write at the same second.
- The `post` and `drain` tools are available on the agent's PATH or at a known location.

## Error Handling

| Situation | Tool | Required Behavior |
|---|---|---|
| Init tool creates inbox dir | `init` | Atomic mkdir; exit 2 if already taken (without --force) |
| Init tool signal write fails | `init` | Exit 3; fatal on init |
| Inbox directory absent on post | `post` | Create directory, then proceed |
| Filename collision detected | `post` | Regenerate nonce and retry |
| `post` tool write fails | `post` | Exit non-zero, message on stderr |
| Claim rename fails (file gone or already claimed) | `drain` | Skip file silently, continue |
| Message file unreadable | `drain` | Report on stderr, move to archive, continue |
| Message parse failure (bad frontmatter or empty body) | `drain` | Report on stderr, move to archive, continue |
| File disappears between enumerate and claim | `drain` | Skip file, treat as claim failure |

## Precedence Rules

P1. Claim-by-rename (R22) is the sole contention control primitive for message files. No
    other mechanism (lock files, advisory locks, semaphores) takes precedence over or
    supplements it.

P2. The signal file has no contention control. Simultaneous overwrites by multiple posters
    are acceptable — last write wins. Because drain is a full sweep, coalescing multiple
    signal events into one drain invocation is safe; no messages are lost.

P3. Filename sort order (R26) takes precedence over signal event arrival order. An event
    that arrives before enumeration is complete does not reorder earlier messages.

P4. All drained messages move to `archive/` regardless of parse outcome. There is no
    separate error state; archive is the only terminal state in v1.

## Don'ts

- Do NOT skip `init` on startup; register your inbox before draining or watching.
- Do NOT call `init` without `--force` on restart; use `--force` to reclaim.
- Do NOT write inbox files directly; always invoke the `post` tool.
- Do NOT invoke `drain` against another agent's inbox; drain your own inbox only.
- Do NOT delete message files from the inbox or archive manually during normal operation.
- Do NOT share one inbox between multiple draining agents in v1.
- Do NOT post a message to your own inbox.
- Do NOT halt inbox processing on a single failing message; log and continue.
- Do NOT assume the inbox is secure; any process with filesystem access can read it (C5).
