Params: `skill_path` (required); `result_file` (required); `spec_path` (optional)

Procedure:
1. Read skill at `skill_path`; determine type: inline or dispatch
2. Locate companion spec — check: `spec_path` if provided → `spec.md` in same dir as `skill_path`. Not found: simple inline (<30 lines) may skip Phase 1; dispatch/complex inline → FAIL immediately.
3. Run Phase 1 → Phase 2 → Phase 3 (stop on first failure) → assign verdict → write report to `result_file`

Phase 1 — Spec Gate: verify companion spec structurally sound before any skill-level checks.
1. Spec exists — `spec.md` must exist (or `spec_path` resolves). Exception: simple inline (<30 lines) → skip phase entirely.
2. Required sections — spec must contain: Purpose, Scope, Definitions, Requirements, Constraints. Missing → FAIL.
3. Normative language — requirements must use enforceable language (must/shall/required). Vague terms in normative sections → FAIL.
4. Internal consistency — no contradictions between sections; no duplicate rules; no normative content in descriptive sections.
5. Completeness — all terms defined; all behavior explicitly stated, not implied.
Phase 1 fail → verdict FAIL, report which check failed, don't proceed to Phase 2.

Phase 2 — Skill Smoke Check: quick structural verification.
1. Classification — "Could someone with no context do this from just the inputs?" Yes → should be dispatch; No → should be inline. Flag misclassification.
2. Inline/dispatch file consistency — File presence is definitive: any allowed dispatch instruction file present (`instructions.txt`, `<name>.md` in skill dir, or instruction file referenced by SKILL.md) → dispatch; none found → inline. If Check #1 and this check disagree, flag the conflict as a finding but do NOT double-fail — note conflict and let auditor decide. If dispatch: verify SKILL.md is a short routing card (structural details in Check #3). If inline: verify SKILL.md contains full procedure. Mismatch between file-system evidence and SKILL.md structure → FAIL.
3. Structure
   Inline: frontmatter (`name`, `description`), direct instructions, self-contained.
   Dispatch: SKILL.md ≤15 lines routing content; instruction file exists and reachable; params typed with required/optional/defaults; output format specified; uses Dispatch agent (isolated), not background agent with host context.
4. Frontmatter — `name` and `description` present and accurate.
5. No duplication — not duplicating existing capability; if similar exists, recommend merge or distinguish.
Phase 2 fail → verdict FAIL, don't proceed to Phase 3.

Phase 3 — Spec Compliance Audit: deep verification SKILL.md faithfully represents spec.
1. Coverage — every normative requirement in spec represented in SKILL.md. Missing → FAIL.
2. No contradictions — SKILL.md mustn't contradict spec. Spec authoritative; SKILL.md subordinate.
3. No unauthorized additions — SKILL.md mustn't introduce normative requirements not in spec.
4. Conciseness — every line affects runtime behavior; no rationale (belongs in spec); no redundant explanations; agent-facing density.
5. Completeness — all runtime instructions present; no implicit assumptions; edge cases addressed or excluded; defaults stated.
6. Breadcrumbs — ends with related skills/topics; references valid (targets exist); no stale references.
7. Cost analysis (dispatch only) — uses Dispatch agent (zero-context isolation); instruction file <500 lines; sub-skills by pointer, not inlined; single dispatch turn when possible.
8. Markdown hygiene — report no errors on every `.md` file in the skill.
9. No dispatch refs in instructions — `instructions.txt` mustn't tell agent to dispatch other skills; subagents can't dispatch, only host agent can; "Related" context refs OK; "run this skill"/"dispatch this" → FAIL; remediation: move dispatch steps to SKILL.md.
10. No spec breadcrumbs in runtime — SKILL.md and `instructions.txt` must not reference the skill's own companion `spec.md` (pointer, breadcrumb, "see spec.md"); compressed runtime is self-contained. Exception: skills operating on specs as input (spec-auditing, skill-auditing) may reference the spec under audit, never their own companion.

Verdict Rules:
PASS: all 3 phases pass.
NEEDS_REVISION: Phase 1 and 2 pass; Phase 3 has 1-2 non-critical issues. List fixes.
FAIL: any phase fails, or Phase 3 has critical issues.
