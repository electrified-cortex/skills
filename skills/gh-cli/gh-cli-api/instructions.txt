GH CLI API — authenticated REST and GraphQL calls via `gh api`. Escape hatch only; prefer domain skills when they cover the operation.

Use for: ops with no dedicated `gh` subcommand; complex GraphQL queries/mutations; bulk/scripted API interactions.

REST read:
```bash
gh api /repos/owner/repo
gh api /user --jq '.login'
```

REST write (POST/PATCH):
```bash
gh api --method POST /repos/owner/repo/issues \
  --field title="title" --field body="body"
gh api --method PATCH /repos/owner/repo/issues/123 \
  --field state="closed"
```

Pagination — multiple reqs; slow on large datasets:
```bash
gh api /user/repos --paginate --jq '.[].name'
```

jq filtering:
```bash
gh api /repos/owner/repo --jq '.stargazers_count'
gh api /repos/owner/repo/issues --jq '[.[] | select(.state=="open") | {number, title}]'
```

GraphQL query:
```bash
gh api graphql -f query='
  { viewer { login repositories(first: 5) { nodes { name } } } }'
```

GraphQL mutation:
```bash
gh api graphql -f query='
  mutation { resolveReviewThread(input: {threadId: "THREAD_ID"}) { thread { isResolved } } }'
```

GitHub Enterprise — add `--hostname` to any call:
```bash
gh api --hostname enterprise.internal /repos/owner/repo
```

Token safety: never pass tokens as CLI args or inline env vars — leaks to shell history. Use `gh auth login` interactively, or set `GH_TOKEN` in shell profile/CI secret before invoking.

Scope: `gh api` REST and `gh api graphql` only. Doesn't replace domain skills, manage GitHub Apps/OAuth Apps, or cover webhook config.
