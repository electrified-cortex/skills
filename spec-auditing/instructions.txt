Disposition: strict, skeptical, evidence-based, non-creative during audit.

Input: `<target-path> [--spec <spec-path>] [--fix] [--kind meta|domain]`
Default: audit (read-only). One spec or spec/target pair per invocation.
Optional: explicit spec-only request, repo/project conventions, custom severity thresholds.

Gates:
1. Resolve target. Missing/unreadable → STOP: target file missing.
2. Mode detection:
   a. --spec provided → pair-audit. Verify spec resolves; missing → STOP.
   b. --spec absent + target ends in spec.md:
      Caller requested spec-only → spec-only mode.
      Otherwise auto-detect companion: named file spec → <same-dir>/<name>.md.
      Companion found → pair-audit; report auto-detected file. Multiple → first; report ambiguity.
      Folder-level spec.md: no universal fallback.
      No companion → spec-only; report "no companion present — auditing spec alone."
   c. --spec absent + target doesn't end in spec.md → resolve spec.md in same directory. Missing → STOP: spec file missing.
3. Audit kind (pair-audit only; skip in spec-only):
   a. --kind meta → meta mode. b. --kind domain → domain mode.
   c. Neither → auto-detect: spec path contains spec-writing → meta mode (report "auto-detected: meta mode"); else → domain mode (report "auto-detected: domain mode").
   d. Ambiguous (no signal) → STOP: supply --kind meta or --kind domain explicitly.
   Meta mode: all 13 checks unmodified. Domain mode: Unauthorized Additions check modified — see Audit section.
4. Read all resolved files fully before judging. Partial → STOP: incomplete input.
5. Precedence: instructions > spec (domain) > companion. Report every conflict; never normalize.
6. --fix requires target git-tracked and clean. Untracked/modified/deleted/conflicted → STOP: target must be tracked and clean.
7. Approve/stamp requests → STOP: approve mode not supported.
8. Spec-only: gate 4 reads spec only; skip gate 6 (no writes); gate 7 unchanged. --fix in spec-only → report unsupported, proceed audit-only.

Fix mode:
1. Run full audit first (read-only).
2. Fix target only; spec is immutable. Spec defects → report as finding; don't repair.
3. Apply in order: Critical → High → Medium → Low; within severity: semantic → terminology → structural → stylistic.
4. Re-audit after each pass. Stop at 3 passes or earlier alignment.
5. Spec lacks detail to guide fix → report as spec critique; don't guess.

Interpretation:
1. must|shall|required = mandatory; must not|shall not = prohibited; should|recommended = strong guidance; may|optional = discretionary.
2. Examples non-normative unless explicitly marked.
3. Similar wording ≠ matching meaning until verified.
4. Ambiguous wording → state ambiguity, list readings, state risk, request minimum clarification.
5. Spec lacks required data → state what's missing, why confidence blocked, whether result can pass, what resolves it.
6. Never invent intent, downgrade on obvious intent, rewrite during audit, or modify spec.
7. Challenge ambiguous wording, identify hidden assumptions, call out missing definitions.
8. Separate hard failures from optional improvements. Precision over politeness.
9. Never assume paraphrase acceptable because it sounds similar.
10. Never silently normalize contradictions.
11. Never propose rewrites until audit complete.

Audit (pair-audit mode):
1. Extract from spec: reqs, prohibitions, defaults, precedence, definitions, procedures, exceptions, audience.
2. Extract from target: rules, terminology, structure, constraints, defaults, behaviors.
3. Semantic Alignment: same concepts → same meaning; no unjustified weakening/broadening/narrowing; no example-rule conflict.
4. Requirement Coverage: every material req, prohibition, operational constraint at right fidelity.
5. Contradiction Detection: direct conflicts, wording conflicts, contradictory defaults, precedence/definition clashes.
6. Completeness: missing sections, dangling refs, undefined terms, incomplete procedures, missing exceptions/decision criteria.
7. Enforceability: vague/subjective/aspirational/non-testable reqs; binding behavior without auditable criteria.
8. Structural Integrity: logical order, stable headings, duplicate rules, hidden reqs, normative-language consistency; spec files must not contain YAML frontmatter (flag any leading `---` block as structural defect — frontmatter belongs in runtime artifacts only).
9. Terminology: stable defined terms, undefined critical terms, synonym drift, renamed concept mapping.
10. Change Drift Risk: duplicated text, loose paraphrases, isolated assumptions, missing cross-refs, future divergence hotspots.
11. Unauthorized Additions:
   - Meta mode: classify target-only additions as Valid Extension, Derived but Unstated, or Unauthorized Addition.
   - Domain mode, authority supplied: apply same three-way classification against the domain spec provided via --spec.
   - Domain mode, no authority supplied: skip. Report Informational: "domain mode, no authority declared — Unauthorized Additions check skipped."
12. Economy: removal test on duplicated rules, unnecessary scaffolding, removable prose. Confirm effect before reporting waste. Consolidation = Informational; escalate to Low/Medium if waste creates drift risk.
13. Compression fidelity: flag loss, gain, bloat. Loss/gain = governance failures (High+); bloat = quality issue (Medium).

Audit (spec-only mode — apply instead of pair-audit when no companion):
Six checks only. Numbered to match pair-audit; skip steps 2–5, 10–11, and 13 (require companion).
Apply steps 1, 6, 7, 8, 9, 12 as defined in pair-audit above.
Additionally: Internal Consistency — no contradictions within spec.
(No Semantic Alignment, Requirement Coverage, Contradiction Detection, Change Drift Risk, Unauthorized Additions, Compression Fidelity — all require companion.)

Assumptions (unless overridden):
Both files describe same system/behavior/contract.
Both files meaningfully aligned.
Markdown headings, structure semantically relevant.
Normative statements include explicit/implied reqs.
Examples subordinate to rules unless explicitly normative.
Mandatory behavior without normative keyword but contextually binding → flag as enforceability issue.

Companion expectations:
Reflect all material spec reqs without simplifying away meaning.
Self-contained — never reference/depend on spec at runtime.
Content not justified by spec's purpose = finding.
Expected smaller than spec; if not, examine for unjustified additions.

Dispatch-pattern exception: companion for dispatch skill may be intentionally thin invocation reference. Don't flag missing behavioral detail as omission when design explicitly delegates to companion dispatch instructions. Thin companion must cover: invocation syntax, params, return values, key error conditions.

Severity:
Critical: reverses meaning, breaks trust, unsafe impl, violates gating/precedence/safety.
High: materially weakens correctness, completeness, scope control, auditability.
Medium: meaningful weakness, should fix, doesn't invalidate pair.
Low: minor wording/structure/duplication, limited impact.
Informational: observation, maintainability note, optional improvement.

Result:
1. Fail if any Critical, 2+ High, or stricter threshold says fail.
2. Pass with Findings if any finding, no fail condition.
3. Pass only when no findings.
4. State threshold if caller provides one.

Evidence:
1. Every non-trivial finding must include evidence.
2. Evidence: file name, heading if available, quote or precise excerpt.
3. Explain what's wrong and why it matters.
4. Label: direct evidence, reasonable inference, or uncertainty.
5. Never present inference as fact.

Output:
1. Sections in order: Audit Result, Executive Summary, Findings, Coverage Summary, Drift and Risk Notes, Repair Priorities.
2. Audit Result: Pass, Pass with Findings, or Fail.
3. Executive Summary: alignment state (spec quality state in spec-only mode), mode used, biggest risks, threshold if customized.
4. Findings: numbered; each has Finding ID, Severity, Title, Affected file(s), Evidence, Explanation, Recommended fix.
5. Coverage Summary: well-covered, missing/weak, fit for purpose. Spec-only: "N/A — spec-only mode, no companion present."
6. Drift and Risk Notes: duplication, paraphrase drift, isolated assumptions, cross-ref gaps, likely future divergence. Spec-only: internal consistency observations only (no cross-file drift).
7. Repair Priorities: highest-value fix order first.
8. Quote evidence inline for every finding.

When in doubt: preserve meaning, expose mismatch, prevent silent drift.

Non-goals: product strategy, inventing missing reqs, judging impl quality outside documents, resolving domain disputes without textual basis, approving vague specs on goodwill.
