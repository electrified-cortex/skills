# dispatch-strategy Spec Audit (Haiku, 2026-04-19)

**Overall Verdict:** NEEDS_REVISION

**Finding Summary:** 1 blocker, 4 major, 1 minor

---

## BLOCKER

**Missing Mitigations for Footguns (B4 Violation)**
- Lines 74–79: Footguns F1–F5 enumerated but no mitigation content provided.
- B4 requires: "Each documented footgun must be followed by the specific dispatch parameter or prompt construction that prevents it."
- Currently each footgun states the problem but not the fix.
- Impact: Spec is aspirational rather than actionable; footguns cannot be avoided without external knowledge.

---

## MAJOR

**R2 & R3 Lack Empirical Anchors (B3, C7 Violations)**
- Lines 57, 59: Core empirical claims ("conversation context is not inherited," "project context IS inherited") stated as fact but lack date/method stamp.
- B3 requires: "Empirical (DATE, METHOD): CLAIM" format for empirical claims.
- C7 requires explicit date anchors so drift is detectable.
- Impact: Readers cannot judge claim staleness or verify method.

**Subagent Type Examples Overstated (C2 Violation)**
- Lines 28–29: Names specific types (`Dispatch`, `general-purpose`, `Plan`, `Explore`, `claude-code-guide`, `statusline-setup`) with signal ("evolves," "examples") but reads as enumeration.
- C2: "must not enumerate the current Claude Code subagent_type list as if it were stable."
- Impact: Likely causes readers to treat list as stable despite hedging language.

**Anti-Pattern Marker Not Defined or Implemented (B5 Violation)**
- Line 11 of Definitions mentions anti-patterns; R11 requires them with "distinct visual marker."
- B5 requires: "use a consistent prefix or block style for anti-patterns so that a skimming agent can identify 'do not do this' content."
- Spec defines the requirement but does not show the marker used in Requirements/Behavior sections.
- Impact: Spec is self-referential; anti-pattern treatment not demonstrated.

**No Anti-Pattern Example Provided (R11 Violation)**
- R11 requires: "at least one explicit anti-pattern example showing a wrong dispatch and explaining why it is wrong."
- Spec lists footguns but does not show a worked wrong-dispatch example with explanation.
- Impact: Requirement stated but unmet in body.

---

## MINOR

**Terminology Inconsistency: "subagent type" vs "subagent_type"**
- Lines 28, 42, 44 use both forms inconsistently.
- Line 28: "subagent_type parameter" (correct for tool parameter).
- Line 42: "Model override" refers to dispatch parameters but does not clarify naming.
- Impact: Low; context clarifies meaning, but spec-writing style requires consistent terminology.

---

## Section-by-Section Assessment

**Purpose, Scope, Definitions:** Sound structure; definitions clear and testable.

**Requirements R1–R10:** Well-formed except R2, R3, R11 noted above. R4–R8 atomic and testable.

**Constraints C1–C8:** Well-articulated; C2 and C7 conflict with current Examples section (see major findings).

**Behavior B1–B6:** Well-defined intent; B4 and B5 not discharged in Requirements.

**Defaults, Assumptions, Error Handling, Precedence:** Sound and enforceable.

**Don'ts DN1–DN10:** Properly exclude off-scope content; DN9 undercut by unanchored R2/R3.

---

## Recommendation

Resolve blockers before dispatch:

1. Add mitigation subsection to each footgun (F1–F5) detailing parameter/prompt fix.
2. Anchor R2 and R3 with empirical date/method: `Empirical (DATE, tested via METHOD): claim.`
3. Reduce subagent_type example count to 1–2 representative types; add explicit "examples only" callout.
4. Define anti-pattern visual marker (e.g., `ANTI-PATTERN:` prefix) and demonstrate with one worked wrong-dispatch example in Behavior or Requirements.

Post-revision re-audit recommended.
