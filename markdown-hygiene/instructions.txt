Fix all markdownlint violations in a .md file. Zero errors is the gate.

Params:
`file_path` (required): absolute path to .md file
`--source X --target Y` (optional): read X, fix, write Y; X untouched, no git check

Modes:
Source→Target: `--source X --target Y` → fix violations, write to Y. Source untouched.
In-place (default): run `git status --porcelain -- <file>`. Empty or `M ` (staged only) → fix directly.
Fallback: unstaged changes, `??` (untracked), or `MM` (staged + additional unstaged) — no `--target` → create `<file>.fixed` alongside.

Procedure:
1. Read file, scan for markdownlint violations using own knowledge.
2. Use available tools (built-in, MCP) — don't install external packages.
3. Fix every violation. Format: `MD<NNN> — description`.
   MD001 — heading level increments
   MD003 — heading style consistency (atx vs setext, uniform across file)
   MD004 — list markers consistent
   MD009 — no trailing spaces
   MD010 — no hard tabs
   MD012 — collapse multiple blank lines to one
   MD022 — blank lines BEFORE and AFTER headings
   MD023 — headings at line beginning, no leading whitespace
   MD024 — no duplicate headings among siblings
   MD025 — single H1 per document
   MD026 — no trailing punctuation in headings
   MD029 — ordered list prefixes consistent
   MD031 — blank lines BEFORE and AFTER fenced code blocks (text immediately before/after ``` triggers this; check every fence)
   MD032 — blank lines around lists
   MD033 — no inline HTML; HTML comments `<!-- ... -->` ARE inline HTML and trigger MD033; don't treat as metadata; exception: HTML inside fenced code blocks is not a violation
   MD034 — bare URLs in angle brackets
   MD040 — language IDs on fenced code blocks
   MD047 — file ends with single newline
   MD055 — consistent table pipe style
   MD056 — equal column count across table rows
   MD058 — blank lines around tables
   MD060 — table pipe spacing
   all other markdownlint rules
4. Write to target (in-place, `.fixed`, or `--target`).
5. Verify: re-run linter on output, confirm zero errors.
6. Cross-check each reported finding by re-reading the cited line. Confirm the rule applies. Do NOT report a violation you cannot point to on a specific line. Hallucinated findings are worse than missed ones.
7. Compute report path via audit-reporting path shape (target-kind from target file path); write report there.

Report Format:
Clean: `CLEAN: <file_path> (0 errors)`
Fixed:
```text
FIXED: <file_path>
- <rule>: <description> (N occurrences)
Errors: <before> → 0
```
Partial:
```text
PARTIAL: <file_path>
Fixed: N errors
Remaining: M errors (manual fix required)
- <rule>: <description> (line N)
```

Verdict mapping for audit-reporting: CLEAN → PASS; FIXED → PASS_WITH_FINDINGS; PARTIAL → NEEDS_REVISION.

Iteration safety: don't re-audit unchanged files. Check git status or mtime; clean+unchanged → return CLEAN early.

Tables — two canonical separator modes, never mix per table:
Mode A (default): 1–3 dashes per cell, pipe-padded. Widths flexible. e.g. `| - | --- | --- |`
Mode B (aligned): dashes match header width. Use only when human readability matters.
Prohibitions:
- no unpadded pipes (`|---|---|`) — insert space on each side of every dash run; pattern-detectable + pattern-fixable; treat as normal violation, not partial
- no extra whitespace in header labels: `|  Header  |` → `| Header |`
- no inconsistent separator widths within one table
- no mixed Mode A + Mode B in same table

Rules:
Fix every violation. Never suppress a rule.
Never change content meaning — only formatting.
Never introduce new violations while fixing others.
Preserve code blocks, frontmatter, technical strings exactly.
Fix would change meaning → report as unfixable.
One file per dispatch.
Use available tools + own knowledge — don't install external packages.
Intentional bad markdown (edge case): violation inside inline code span (`...`) or clearly content (e.g. tutorial demonstrating wrong syntax marked as such) → preserve as-is, report PARTIAL with line. Don't fix content meant to demonstrate a violation.
