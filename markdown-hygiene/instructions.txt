Detect and optionally fix markdownlint violations. Zero errors is the gate.

Params:
`file_path` (required): absolute path to .md file
`--fix` (optional): apply fixes after detecting; without this, detect only — file never modified
`--source X --target Y` (optional): read X, fix, write Y; X untouched, no git check; implies `--fix`
`--ignore <RULE>[,<RULE>...]` (optional): skip rule codes — not scanned, not flagged, not fixed
`--force` (optional): bypass cache HIT, re-execute even if cached record exists
Adaptive MD041: first non-blank line is `---` -> auto-suppress MD041, no flag needed

Modes:
Detect (default, no `--fix`): read-only, no git-state check; always runs; produces one detect record.
Fix (with `--fix`): `git status --porcelain -- <file>` -> proceed only if output empty or first char `M` second char space; else `ERROR: target_dirty: stage changes and re-run with --fix; cached detect record carries forward (no rescan)`, stop. Writes second record at post-fix hash.
Source->Target: `--source X --target Y` -> read X, fix, write Y; source untouched; implies `--fix`

Procedure:

Executor, not planner. Run each step's tool calls. Don't summarize, don't produce "fixed version" inline. Don't stop early. Dispatch incomplete unless record file written and absolute path output.

Two passes: detect first (always), fix second (only if `--fix` or `--source/--target` passed).

Pass 1 — Detect:

1. Read target (Read tool). Guard: unreadable or not `.md` -> `ERROR: <reason>`, stop. Adaptive MD041: first non-blank line `---` -> add MD041 to suppressed set.
2. Run: `git hash-object <file_path>` (Bash tool). Save 40-char result as `<hash>`.
3. Cache check: `test -f <repo-root>/.hash-record/<hash[0:2]>/<hash>/markdown-hygiene/<model>.md` (Bash). Skip if `--force` passed.
   - Exists: read its `result:` frontmatter field -> if `pass` output `CLEAN`; if `findings` output `findings: <repo-root>/.hash-record/<hash[0:2]>/<hash>/markdown-hygiene/<model>.md` — stop (cache HIT).
   - Missing: cache MISS. Save `<repo-root>/.hash-record/<hash[0:2]>/<hash>/markdown-hygiene` as `<detect_cache_dir>` (no trailing slash). Continue.
4. Scan for markdownlint violations. Prefer `markdownlint` CLI (see co-located `tooling.md`); else use rule knowledge. Skip rules in `--ignore` or adaptive suppressed set. Cross-check each finding against actual line before recording — drop any finding not pointable to specific verified line.
   Per-rule anchors:
   - MD001: heading levels must increment by one (H1->H3 = violation)
   - MD003: heading style must be consistent (atx `#` vs setext `===`/`---`); flag any heading differing from first
   - MD004: list markers must be consistent (`-`, `*`, `+`); flag any item using different marker than first in its list
   - MD009: trailing spaces; flag any line ending with 1+ spaces (except two-space line-break)
   - MD010: hard tabs; flag any literal tab in non-code-block content
   - MD012: multiple consecutive blank lines; flag runs of 2+
   - MD022: headings must be preceded AND followed by blank line (except file start/end)
   - MD023: headings must not have leading whitespace (line must begin with `#`)
   - MD024: duplicate heading text among siblings; flag if two same-level same-parent headings share identical text
   - MD025: only one H1 per file; flag every H1 after first
   - MD026: headings must not end with punctuation (`.`, `!`, `?`, `:`, `,`, `;`)
   - MD029: ordered list prefixes must be consistent: all `1.` or strictly incrementing
   - MD031: fenced code blocks must be preceded AND followed by blank line
   - MD032: lists must be preceded AND followed by blank line
   - MD033: no inline HTML; flag `<tag>`, `</tag>`, `<!-- comment -->` outside fenced code blocks
   - MD034: bare URLs; flag any `http://` or `https://` not inside angle brackets, backticks, or a link
   - MD040: fenced code blocks must have language identifier (` ```python `, not bare ` ``` `)
   - MD041: first non-blank line must be H1 (suppressed when frontmatter detected or `--ignore MD041`)
   - MD047: file must end with exactly one newline character
   - MD055: table pipe style must be consistent across all rows
   - MD056: all rows in a table must have same number of cells
   - MD058: tables must be preceded AND followed by blank line
   - MD060: table cell separators must have space on each side of dash run (`| --- |` not `|---|`)
5. Write detect record at `<detect_cache_dir>/<model>.md`:
   - Run: `mkdir -p <detect_cache_dir>` (Bash tool).
   - Frontmatter: `hash`, `file_path`, `operation_kind: markdown-hygiene`, `model: <your-model-id>`, `result: pass` (no violations) or `result: findings` (violations exist).
   - Body: `# Result\n\nCLEAN` or `# Result\n\nFINDINGS\n\n- <list>`. FINDINGS: each entry = two lines: `MD0XX line N: <description>` then indented `Fix: <imperative>`. `Fix:` MUST be complete standalone instruction — Pass 2 applies literally without rule knowledge.
   - No violations (CLEAN): output `CLEAN`, stop.
   - Violations + `--fix` not passed: output `findings: <detect_cache_dir>/<model>.md`, stop.

Pass 2 — Fix (only when `--fix` present and violations exist):

6. All violations unfixable (manual-only, e.g. MD040 no language to infer) -> output `findings: <detect_record_path>` (no second record), stop.
7. Read detect record (Read tool). Extract every `Fix:` line from FINDINGS body. Apply each literally to target (Edit/Write tool). No re-scan, no rule knowledge required. File on disk MUST contain all applied fixes after this step.
8. Compute new hash: `git hash-object <fixed-file-path>`. Save as `<fix_hash>`. Set `<fix_cache_dir>` = `<repo-root>/.hash-record/<fix_hash[0:2]>/<fix_hash>/markdown-hygiene` (no trailing slash).
9. Write fix record at `<fix_cache_dir>/<model>.md`:
   - Run: `mkdir -p <fix_cache_dir>` (Bash tool).
   - Frontmatter: `hash: <fix_hash>`, `file_path`, `operation_kind: markdown-hygiene`, `model: <your-model-id>`, `result: pass` (all fixed) or `result: findings` (some remain).
   - Body: `# Result\n\nFIXED\n\n- <list>` or `# Result\n\nPARTIAL\n\n...`.
   - All fixed (FIXED): output `CLEAN`, stop. Some remain (PARTIAL): output `findings: <fix_cache_dir>/<model>.md`, stop.

Dispatch return (one line, always):
Clean (no violations OR all fixed): `CLEAN`
Findings (violations remain — detect-only or PARTIAL): `findings: <absolute-path-to-record.md>`
Pre-write failure: `ERROR: <reason>`

Report body — minimum info not in frontmatter. Frontmatter holds `file_path`, `result`, `hash`, `model`. Never duplicate `file_path` in body. Body always opens with `# Result` H1.

CLEAN:
```text
# Result

CLEAN
```
FINDINGS (violations found, no `--fix`) — each entry: `MD0XX line N: <description>` then indented `Fix: <imperative>` (Pass 2 applies Fix: literally):
```text
# Result

FINDINGS

- MD022 line 7: blank line missing before heading "Configuration"
  Fix: insert blank line before line 7
- MD047: file lacks trailing newline
  Fix: append a single newline at end of file
```
FIXED (all resolved):
```text
# Result

FIXED

- MD022: blank lines around heading (1)
- MD047: trailing newline (1)
```
PARTIAL (some remain unfixable):
```text
# Result

PARTIAL

Fixed:
- MD022 (1)

Remaining (manual):
- MD040 line 14: missing language ID
```

Verdict mapping: CLEAN -> pass; FIXED -> pass; FINDINGS -> findings; PARTIAL -> findings; unrecoverable -> error.

Tables — two canonical separator modes, never mix per table:
Mode A (default): 1-3 dashes per cell, pipe-padded; widths flexible. e.g. `| - | --- | --- |`
Mode B (aligned): dashes match header width; use only when human readability matters.
Prohibitions: no unpadded pipes (`|---|---|`) — pattern-detectable + pattern-fixable, treat as normal violation; no extra whitespace in header labels (`|  Header  |` -> `| Header |`); no inconsistent separator widths within one table; no mixed Mode A + Mode B in same table.

Rules:
Fix every violation. Never suppress unless in `--ignore`.
Never change content meaning — only formatting.
Never introduce new violations while fixing.
Preserve code blocks, frontmatter, technical strings exactly.
Fix would change meaning -> report as unfixable.
One file per dispatch.
Use available tools + own knowledge — don't install or invoke external packages.
Intentional bad markdown: violation inside inline code span or clearly demonstrating wrong syntax -> preserve as-is, report PARTIAL with line. Don't fix content meant to demonstrate a violation.
